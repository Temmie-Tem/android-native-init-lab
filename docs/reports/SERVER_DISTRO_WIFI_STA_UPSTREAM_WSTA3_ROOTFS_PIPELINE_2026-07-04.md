# Server-Distro Wi-Fi STA Upstream WSTA3 Rootfs Pipeline

- Date: 2026-07-04 KST
- Unit: WSTA3 Debian appliance rootfs/tarball guard.
- Scope: source/static validation plus host-only private rootfs build.
- Device mutation: none. No flash, no reboot, no userdata write, no Wi-Fi association, no DHCP, no ping, no public tunnel start.

## Verdict

The WSTA helper is now enforced in the actual D3/D4 userdata appliance pipeline, not only in the generic Debian
rootfs builder.

Before this unit, the fixed D4C default rootfs source did not contain:

- `/usr/local/bin/a90-dpublic-wifi-sta`
- `/etc/a90-dpublic`

That meant WSTA1 source support existed, but an old D3 sysvinit rootfs could still be staged to userdata without
the Debian STA helper.  This unit closes that gap.

## Change

`workspace/public/src/scripts/server-distro/prepare_d3_sysvinit_rootfs.py` now stages:

- source: `workspace/public/src/scripts/server-distro/a90_dpublic_wifi_sta.sh`
- target: `/usr/local/bin/a90-dpublic-wifi-sta`
- mode: `0755`
- config directory: `/etc/a90-dpublic`

The D3 stage marker also records:

- `wifi-sta=opt-in via /etc/a90-dpublic/wifi-sta-enable, private config not included`
- `wifi-sta-helper=/usr/local/bin/a90-dpublic-wifi-sta`

`workspace/public/src/scripts/server-distro/prepare_d4c_userdata_rootfs_tarball.py` now rejects a rootfs that
does not include an executable STA helper and the `/etc/a90-dpublic` config directory.  Its tarball verification
also requires the helper entry.
