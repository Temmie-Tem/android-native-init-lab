# S22+ FYG8 R3 Carrier And Static Checker Audit

Date: 2026-07-12 KST
Target: `SM-S906N/g0q/S906NKSS7FYG8`
Scope: host-only read-only analysis; no artifact generation, repack, AP build,
device contact, or flash

## Verdict

The former R3 requirement that a no-change MagiskBoot repack of the exact FYG8
stock `boot.img` be byte-identical is **source-disproved**.

The stock image has a 528-byte Samsung post-payload signer area beginning with
`SEANDROIDENFORCE` and `SignerVer02`. MagiskBoot v30.7 recognizes the first
16-byte marker but writes back only that marker. It does not preserve the
remaining 512-byte signer record. It also updates the AVB footer's
`original_image_size` while copying the original vbmeta blob. Therefore an
exact-stock no-change repack is expected to differ even when the kernel and
ramdisk are unchanged.

This does not prove whether S-Boot requires, ignores, or conditionally checks
the signer record on the unlocked device. It does prove that a direct rebuilt
kernel insertion into the exact stock container would mix two hypotheses:

1. rebuilt kernel viability; and
2. acceptance of a present-but-stale Samsung signer record.

R3 is therefore corrected to a two-rung differential:

1. **R3C0 carrier control:** stock kernel plus stock ramdisk in the exact,
   statically characterized MagiskBoot-normalized container;
2. **R3C1 kernel differential:** the byte-identical R3C0 control with only the
   kernel region replaced by the exact R2 Image.

R3C1 is not eligible for live use unless R3C0 boots and rolls back in a separate
one-shot session. No image or AP was generated by this audit.

## Stock Boot V4 Geometry

The exact stock image decomposes into the following regions:

| # | Region | Byte range | Length | Required state |
|---:|---|---:|---:|---|
| 1 | header page | `[0, 4096)` | 4,096 | exact stock |
| 2 | kernel | `[4096, 41495040)` | 41,490,944 | exact stock or exact R2 by rung |
| 3 | kernel alignment | `[41495040, 41496576)` | 1,536 | exact stock |
| 4 | ramdisk | `[41496576, 43475543)` | 1,978,967 | exact stock |
| 5 | ramdisk alignment | `[43475543, 43479040)` | 3,497 | exact stock |
| 6 | GKI boot signature field | `[43479040, 43483136)` | 4,096 | exact stock; all zero |
| 7 | Samsung signer area | `[43483136, 43483664)` | 528 | stock record or normalized form |
| 8 | AVB pre-padding | `[43483664, 43487232)` | 3,568 | all zero |
| 9 | AVB vbmeta | `[43487232, 43489344)` | 2,112 | exact stock |
| 10 | final padding | `[43489344, 100663232)` | 57,173,888 | all zero |
| 11 | AVB footer | `[100663232, 100663296)` | 64 | stock or normalized control value |

The stock footer records `original_image_size=43483664`, which includes the
complete 528-byte Samsung signer area. The R2 Image has exactly the same file
length and the same ARM64 image-header fields as the stock kernel. This closes
layout capacity only; it does not prove boot acceptance.

The stock kernel, ramdisk, and boot-signature slices independently match the
files emitted by the pinned parser. Stock `avbtool verify_image` passes fully.

## Samsung Signer Finding

The 528-byte area after the GKI signature is not padding. It starts with:

```text
SEANDROIDENFORCE
SignerVer02
99315284R
S906NKSS7FYG8
```

Its SHA256 is
`13c2a41e4b47b0e7173d054ddff148bfcc195e639c7bca137753c03e8dfedc36`.
The known-booting Magisk image retains only the 16-byte
`SEANDROIDENFORCE` marker at this location. Its footer consequently records
`original_image_size=used_end+16`, while its vbmeta bytes remain identical to
stock and its boot-payload digest is stale.

The exact Magisk v30.7 source confirms the transformation:

- `native/src/boot/bootimg.cpp:583-585` detects the 16-byte Samsung marker;
- `native/src/boot/bootimg.cpp:919-920` writes only 16 bytes on repack;
- `native/src/boot/bootimg.cpp:1020` updates AVB footer
  `original_image_size`;
- the copied vbmeta hash descriptor is not regenerated.

Upstream references:

- Magisk repository: <https://github.com/topjohnwu/Magisk>
- exact v30.7 boot-image implementation:
  <https://github.com/topjohnwu/Magisk/blob/v30.7/native/src/boot/bootimg.cpp>
- v30.7 release: <https://github.com/topjohnwu/Magisk/releases/tag/v30.7>

The repository's DTBO builder already treats a separate 512-byte
`SignerVer02` trailer as signed structure rather than zero padding at
`workspace/public/src/scripts/revalidation/build_s22plus_v3435_ramoops_console_dtbo.py:524`.
The boot checker must adopt the same fail-closed principle.

## Corrected R3 Differential

### R3C0 - Normalized Carrier Control

R3C0 must contain the stock kernel and stock ramdisk. Relative to exact stock,
the only allowed byte differences are:

1. the Samsung signer area becomes exactly the 16-byte
   `SEANDROIDENFORCE` marker followed by zero bytes through the original
   528-byte region; and
2. the AVB footer `original_image_size` changes from `43483664` to
   `43483152` (`used_end + 16`).

For this fixed geometry, `vbmeta_offset` must remain `43487232`. The header,
kernel, alignment, ramdisk, GKI signature, vbmeta, and all other padding must
remain byte-identical to stock.

R3C0 is a carrier-acceptance control. It does not test the rebuilt kernel.

### R3C1 - Kernel-Only Differential

R3C1 must be derived from the exact, independently pinned R3C0 raw boot bytes
by replacing only `[4096, 41495040)` with the exact R2 Image. It must not run a
second semantic repack.

R3C1 versus R3C0 must differ only in the kernel region. Every non-kernel byte,
including the normalized Samsung marker area, stale vbmeta blob, and footer,
must be byte-identical. This makes a failed R3C1 boot attributable to the
rebuilt-kernel path after R3C0 proves the carrier path.

R3C0 and R3C1 require separate attended one-shot live sessions and separate
eight-event timelines. R3C1 remains blocked unless R3C0 reaches its declared
Android milestone and completes rollback.

## Static Checker Contract

The next host-only implementation should be a checker first. Artifact creation
remains a later policy unit. The checker must:

1. pin all input sizes and SHA256 values above plus the R1/R2 PASS fields;
2. parse boot v4 with bounded offsets and reject overlap, truncation, trailing
   ambiguity, or unexpected header fields;
3. model all 11 regions explicitly instead of calling the complete post-payload
   area padding;
4. recognize exactly the stock 528-byte signer record or exactly the normalized
   16-byte marker form; reject every mixed or unknown form;
5. prove the R3C0 allowed-delta set and the R3C1 kernel-only delta set at byte
   range granularity;
6. require exact stock ramdisk bytes and reject Magisk/native-init additions;
7. report stock AVB full verification separately from normalized/candidate
   vbmeta signature verification and expected stale payload-digest failure;
8. validate an AP as one regular `boot.img.lz4` member with no duplicate name,
   PAX override, path traversal, link, device, or extra tar member;
9. validate the exact 41-byte MD5 trailer, tar-prefix MD5, 512-byte tar
   alignment, terminal zero blocks, one strict LZ4 frame, no trailing frame
   data, and the decompressed raw boot hash and size;
10. emit no image, AP, USB access, Odin call, or device action.

Existing reusable code:

- `s22plus_fyg8_magisk_boot_audit.py`: bounded boot-v3/v4 parser, AVB footer,
  CPIO newc parser, and diff utilities;
- `build_s22plus_direct_p3_boot.py`: deterministic tar and store-only LZ4
  patterns, but its decoder requires stricter bounds and trailing-data checks;
- `s22plus_m3_observable_live_gate.py`: AP identity pattern, but its tar check
  is name-only and is insufficient for this checker.

The current audit parser's generic `trailing_padding` classification is not
safe for exact stock R3 and must not be reused unchanged.

## Threat Model And Fail-Closed Cases

The checker must reject:

- zero or multiple matching input artifacts;
- a stock image whose signer record, vbmeta, footer, or region geometry differs
  from the pins;
- a normalized control containing any nonzero residual signer bytes;
- a candidate differing outside the kernel region;
- any candidate kernel size or ARM64 header mismatch;
- a changed ramdisk, bootconfig, cmdline, GKI signature, vbmeta, DTBO, recovery,
  or vendor_boot input;
- an AP with ambiguous tar semantics or noncanonical LZ4 contents;
- any attempt to interpret a static PASS as live authorization.

## Unresolved Evidence

Static inspection of FYG8 ABL/UEFI images found `SignerVer02` records on their
own signed-image tails but did not locate a consumer routine that proves how
the boot image signer record is enforced. Therefore all of the following remain
unproved:

- S-Boot ignores the boot signer record in unlocked/orange state;
- S-Boot validates it before AVB state handling;
- S-Boot accepts the normalized 16-byte form because Magisk uses it;
- acceptance depends on another state not represented in the boot image.

R3C0 is the minimal live discriminator for those unresolved alternatives. A
deeper bootloader consumer RE can proceed in parallel, but it cannot replace
the carrier control unless it proves the exact FYG8 branch and state.

## Independent Review

A persistent Claude Opus adversarial review received the exact geometry,
hashes, Magisk v30.7 source behavior, known-Magisk comparison, and existing
R3 contract. It agreed that the old no-change premise is disproved, selected
the two-rung normalized-control architecture as the cleanest differential, and
returned GO for checker-source implementation only. It returned NO-GO for
artifact generation, AP packaging, device contact, or flash without a fresh
SHA-pinned exception.
