# S22+ — The Stock modules.dep Is the USB Bring-Up Answer: Our Closure Is Dependency-Incomplete (Operator Finding, 2026-07-08)

Operator (Claude) host-only firmware analysis. No device action. This answers
the operator-user's question "is the USB driving method / a design or reference
already in the firmware?" — **yes**: the vendor's own `modules.load` +
`modules.dep` in the FYG8 vendor_boot ramdisk are the authoritative bring-up
recipe, and they prove why M25/M26 died in modules 1–24.

## Source of truth

From the extracted FYG8 vendor_boot ramdisk
(`unpack-vendor-boot/vendor_ramdisk00`, also mirrored under the M25 build's
`build/vendor_ramdisk_metadata/lib/modules/`):

- `modules.load` — 140 modules, the stock first-stage load order.
- `modules.dep` — depmod-generated hard **symbol** dependency graph.
- `modules.blocklist` — vendor's own runtime blocklist.

`modules.dep` is generated by `depmod` from real inter-module symbol
dependencies, so a listed dependency is a **hard** requirement: the depending
module cannot resolve its symbols (and `insmod` fails, or the driver half-inits
and a downstream consumer hangs) unless the dependency is already loaded.

## Finding: the M25/M26 40-module closure omits hard dependencies

The M25 builder computes its closure as
"HS-only DTS-seed transitive `modules.dep` closure, **minus** a reset/anomaly
blocklist" (`build_s22plus_m25_hs_only_usb2_acm.py`,
`HS_ONLY_BLOCKLIST = m23.RESET_ANOMALY_BLOCKLIST | {...}`). The closure code
(lines ~126–147) *actively cuts dependency edges that point into the blocklist*
and even records them as `blocked_dependency_edges` in the manifest — then ships
the cut closure anyway.

A completeness check of the 40-module list against `modules.dep` shows the cut
edges are core-substrate suppliers, not peripheral debug:

```text
 #1 clk-rpmh.ko            MISSING: minidump.ko sec_debug.ko
 #3 icc-rpmh.ko            MISSING: minidump.ko sec_debug.ko
 #4 qcom_ipc_logging.ko    MISSING: minidump.ko sec_debug.ko
 #5 rpmh-regulator.ko      MISSING: minidump.ko sec_debug.ko
#11 icc-bcm-voter.ko       MISSING: minidump.ko sec_debug.ko
#14 qnoc-waipio.ko         MISSING: minidump.ko sec_debug.ko
#18 qcom_rpmh.ko           MISSING: minidump.ko sec_debug.ko
#24 socinfo.ko             MISSING: sec_debug.ko
#26 phy-msm-snps-hs.ko     MISSING: abc.ko
#27 phy-msm-snps-eusb2.ko  MISSING: abc.ko
#28 dwc3-msm.ko            MISSING: abc.ko minidump.ko sec_debug.ko
#29 usb_f_ss_mon_gadget.ko MISSING: abc.ko
#37 usb_typec_manager.ko   MISSING: abc.ko
```

`clk-rpmh.ko` is module **#1** and is the RPMh clock-tree root; it declares
`minidump.ko` and `sec_debug.ko` as dependencies. Under a flat-list bare init
those two are never loaded, so the very first module fails to resolve symbols
and the entire clock/regulator/interconnect substrate that everything else needs
never comes up. This is exactly the M26 localization: **P00 (0 modules) HIT,
P24 (first 24 modules) NO-HIT — the fault is inside modules 1–24, upstream of
USB.**

## The fix is surgical and firmware-mandated: add exactly 3 modules

The three cut dependencies are cheap to satisfy and, critically, **do not drag
in the petting watchdog**:

```text
sec_debug.ko  needs: (nothing)
minidump.ko   needs: smem.ko        (already list #23)
abc.ko        needs: sec_class.ko   (already list #21)
```

And the petting watchdog is provably separable:

- `qcom_wdt_core.ko` / `gh_virt_wdt.ko` are **not** a dependency of any module in
  our substrate. The modules that depend on `qcom_wdt_core` are
  `max77705_charger`, `mhi*`, `dhd`, `sec_qc_*`, etc. — none of which are in the
  40-list.

So adding `sec_debug.ko` + `minidump.ko` + `abc.ko` makes the closure
dependency-complete while keeping `qcom_wdt_core`/`gh_virt_wdt` excluded — the
fault-avoidance (no pet-or-bite watchdog) is preserved.

Bonus: `sec_debug.ko` + `minidump.ko` are exactly the modules that register
Samsung's crash-context capture. Loading them as substrate deps may re-open the
`reset_summary`/`minidump` observability channel that earlier probes found empty
(they were empty in part because these modules were never loaded under bare
init).

## Why this supersedes blind prefix bisection

If the closure is broken at module #1, **every** non-empty prefix (P08…P24) is
broken for the same reason, so a live prefix sweep mostly re-measures the same
missing-dependency failure and spends attended flashes doing it. The firmware
already hands us the corrected recipe. The right M27 is:

1. **Host-only:** rebuild the closure dependency-complete — keep the
   `RESET_ANOMALY` blocklist for the *petting watchdog* and true anomaly-reset
   modules, but **stop cutting `modules.dep` edges that are hard symbol
   suppliers**; specifically re-include `sec_debug.ko`, `minidump.ko`, `abc.ko`
   (and audit any other `blocked_dependency_edges` in the manifest the same way).
2. Load in `modules.dep`-respecting order (dependencies before dependents), not a
   flat DTS-seed order, so no module is insmod'd before its suppliers.
3. Keep the M25 DTBO high-speed cap and QMP exclusion.
4. **Then** one fresh SHA-pinned live gate. Two clean success shapes are now
   possible: a park candidate that finally enumerates `/dev/ttyGS0`, or — if a
   pure survivability check is preferred first — a `reboot(download)` beacon
   after the dependency-complete substrate load (proving 1–24 now survives).

Only after 1–24 survives should the USB tail (25–40) + configfs ACM be trusted,
and only then does the earlier "DTBO still carries the ssphy phandle" question
become relevant (it is downstream of this).

## General lesson (do-not-repeat)

Computing a module set as "`modules.dep` transitive closure **minus** a safety
blocklist" is unsound when the blocklist intersects hard symbol suppliers. On
Samsung GKI the `sec_debug`/`minidump`/`abc` debug-infra modules are woven into
the core power/clock/interconnect substrate as real symbol dependencies, not
optional add-ons. Blocklist only leaf/consumer modules (the petting watchdog,
true reset-anomaly handlers), never a module that appears on the right-hand side
of a `modules.dep` edge for something you intend to keep.

## Discipline

Host-only analysis; the rebuild is host-only until a fresh SHA-pinned boot-only
(or boot+dtbo) `AGENTS.md` exception + attended ack is granted. No forbidden
partitions, no PMIC/power writes, redacted logs (strip serials). Device stays on
the clean rooted Android baseline; v2321 is the A90 rollback target (unrelated).
